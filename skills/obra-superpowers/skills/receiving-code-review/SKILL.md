# 接收代码审查 (Receiving Code Review)

📦 **仓库**: `yanghao1143/chiclaude-skills`
🔥 **安装量**: 6.4K
🔗 **出处**: https://github.com/yanghao1143/chiclaude-skills

---

## 技能简介

接收代码审查反馈时的最佳实践，在实施建议之前进行技术验证，而不是表演性同意或盲目实施。

**适用场景**：接收代码审查反馈时，在实施建议之前，特别是当反馈似乎不清楚或技术上存在问题时。

---

## 核心原则

**核心原则**：验证后再实施。先问再假设。技术正确性胜过社交舒适。

---

## 响应模式

```
当收到代码审查反馈时：

1. 阅读：完整阅读反馈，不反应
2. 理解：用自己的话重述要求（或询问）
3. 验证：对照代码库实际情况检查
4. 评估：对这个代码库来说技术上是合理的吗？
5. 回应：技术确认或有理由的反对
6. 实施：一次一项，每项测试
```

---

## 禁止的响应

**永远不要**：
- "你完全正确！"（明确的违规）
- "好建议！" / "很好的反馈！"（表演性的）
- "我现在就实施"（验证之前）

**应该做**：
- 重述技术要求
- 提出澄清问题
- 如果是错的，用技术推理反驳
- 直接开始工作（行动 > 语言）

---

## 处理不清楚的反馈

```
如果有任何项目不清楚：
  停止 - 暂不实施任何东西
  询问对不清楚项目的澄清

原因：项目可能相关联。部分理解 = 错误实施。
```

---

## 来源特定处理

### 来自用户/合作伙伴
- **可信** - 理解后实施
- **仍有疑问** 如果范围不清楚
- **无表演性同意**
- **直接行动**或技术确认

### 来自外部审查者
```
实施前：
  1. 检查：对这个代码库技术正确吗？
  2. 检查：会破坏现有功能吗？
  3. 检查：当前实现有原因吗？
  4. 检查：在所有平台/版本上工作吗？
  5. 检查：审查者了解完整上下文吗？

如果建议似乎是错的：
  用技术推理反驳

如果无法轻易验证：
  说明："没有[X]我无法验证。我应该[调查/询问/继续]？"

如果与用户之前的决定冲突：
  先停下来与用户讨论
```

---

## 实施顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的项目
  2. 然后按此顺序实施：
     - 阻塞问题（崩溃、安全）
     - 简单修复（错别字、导入）
     - 复杂修复（重构、逻辑）
  3. 逐个测试每个修复
  4. 验证无回归
```

---

## 何时反驳

**在以下情况下反驳**：
- 建议会破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对这个技术栈技术上不正确
- 存在遗留/兼容性原因
- 与用户的架构决策冲突

**如何反驳**：
- 使用技术推理，不是防御性
- 提出具体问题
- 引用工作中的测试/代码
- 如果是架构问题，让用户参与

---

## 确认正确的反馈

当反馈确实是正确的：

```
✅ "已修复。[简要描述更改了什么]"
✅ "好发现 - [具体问题]。已在[位置]修复。"
✅ [直接修复并在代码中展示]

❌ "你完全正确！"
❌ "好建议！"
❌ "谢谢你发现那个！"
❌ "感谢[任何事]"
❌ 任何感激表达
```

**为什么不要感谢**：行动胜于言。直接修复。代码本身表明你听到了反馈。

---

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述要求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 批量实施而不测试 | 一次一项，每项测试 |
| 假设审查者是对的 | 检查是否会破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实施 | 先澄清所有项目 |
| 无法验证但还是继续 | 说明限制，询问方向 |
